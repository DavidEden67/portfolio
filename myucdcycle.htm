My UCD Cycle

Product development has changed substantially over the past few years with the adoption of Agile Methodology. I have been refining my own User-Centered Design cycle to help Agile teams become more successful at delivering products that both the product owners and the end users find useful as well as easy to use. The User Experience discipline can adapt very well to Agile development by conducting early user research during the Planning Phase to move design thinking further upstream, and further, by validating design concepts with users during the Design Phase before development and coding begin. I have been able to contribute to the entire product lifecycle by combining both my research and my design skills.

While not every product goes through the entire UCD or Agile cycle, any usability improvements are better than none. Awareness of the essential UCD activities shown below has helped me become more proactive by predicting where usability issues are likely to occur if the Agile teams skipped one or a few important steps.

1. Problem Statements

Problem statements provide the foundation for the project: they form the mission for the project and must be clearly communicated to all the team members. Problem statements should be continually referenced throughout the duration of the project to help the team stay focused and on track. It is important to document these statements not only from the perspective of the Product Owner and stakeholders, but also from the perspective of end users as to what would make the product a success when it is released to them. During the Agile development problem statements help prioritize features for each sprint, and ensure that the product backlog is addressed accordingly. 

2. Personas

Understanding the end user is the first step toward creating a good design. I conduct user interviews and contextual inquiries to better understand the users and their technical abilities; their expectations of the product, and their pain points. This is particularly important in role-based software products where multiple players interact and each adds a piece to the final output. I create Personas as one of the deliverables at this stage since it helps the team better understand who will be using the system and why. For example, a new HR system that I designed at IBM included the Manager, Employee, Human Resources, IT Administrators, and Legal as the main personas. Creating personas for each type of user; their interactions with the proposed system, and highlighting their pain points created better awareness of a holistic design before the development phase began.

3. User stories & Journeys

Understanding user motivations and documenting user goals create the blueprint for the user requirements documents. User stories and journeys are high level but provide valuable insights for understanding the product features that would motivate the users to use a product. For example, purchasing, unboxing, installing, and using a printer are all parts of the user journey, and while I was at HP I was surprised how each part of the journey could be improved based on user feedback. For example, home and business users have different goals and expectations during the purchasing phase of the journey, and as such a more role-based experience was developed to address this pain point. The same is true for the IT departments who go through a very different user journey for installing printers than home users.

4. Use Cases

While user stories and user goals describe the user needs at a high level, those goals must be broken down into specific tasks and steps by UX designers in order to help users accomplish their goals. Further, use cases must not only include the user goals, but they must also include the business requirements as well as the technical requirements.

The use cases must demonstrate how the user steps through clear paths to accomplish each goal, and what the possible error conditions are and how to help the users out of those errors successfully. Some use cases apply to all users (e.g., user login) while some apply to only certain personas, and some should be given higher priority than others in the design. I typically deliver a Flowchart or a User Journey for each use case that demonstrates how users interact with the system (and each other) to successfully accomplish each goal. I take the best case scenario, the worst case scenario, and several other common uses cases in between the two levels of difficulty, and ensure that my design can handle the entire range of complexity when I translate the use cases into prototypes..


5. Wireframing and Interaction Design

I am skillful at prototyping my designs for the environment in which the product will be used (Desktop, Mobile, and Web) using appropriate platforms such as Bootstrap, CS3, and jQuery. A clear and detailed interactive design is vital for usability inspections and for getting approval from product owners. My interactive prototypes were also key in communicating with offshore developments teams who could better understand the intent of the design and the look & feel of it.

6. Usability Inspections

I use a variety of Usability Inspection techniques to evaluate my designs (depending on the Agile development stage). These include Cognitive Walkthroughs, Discount Usability Evaluations (Heuristics), Pluralistic Walkthroughs, and Usability Testing (both Formative and Summative). The strengths of my rapid prototyping skills have been key for conducting early user evaluations to ensure that not only the product requirements are met from the user and business perspective, but that the design itself is intuitive, learnable, and will have room to grow and adapt in the future. These usability inspections help catch many issues early on which will substantially reduce rework by development teams later in the Agile build phase.

7. User Interface Specification

Upon the approval of final design, I write clear functional specs for developers since relying on interactive prototypes alone is not sufficient in order to write detailed code. I define the behavior of the interface through redlines and control-action-response tables for developers, and I understand code well enough to be able to communicate with developers effectively when I describe the software behavior in terms of CSS and HTML attributes and events.

8. User Acceptance Testing

It is critical for UX Design to be involved during UAT when the actual software is being tested, and analyze direct feedback from users to determine if there are any Severity 1 issues with the product that should prevent it from being released. I coordinate with the QA to make sure that we can detect, track, prioritize, and address usability issues that are discovered during UAT.

9. Post-Release User Feedback

User feedback after each product release should be monitored in order discover new user requirements and to address any usability issues with the current release.
